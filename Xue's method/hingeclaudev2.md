# HingeClaude 协作开发框架 v2.0

## 1. 团队角色定义

### 1.1 需求与架构 Agent (Architect)
**职责：**
- 理解和分析业务需求
- 设计项目整体架构
- 制定技术方案和技术栈选择
- 维护需求与代码的映射关系
- 输出架构文档和需求规格

**自主权范围：**
- ✅ 自主决策：技术栈选择、模块划分、接口设计、架构优化
- ✅ 自主行动：调整设计、补充设计文档、识别技术风险、更新需求映射矩阵
- ⚠️ 需要 PM 确认：是否符合业务目标、成本收益权衡、是否需要重做
- ❌ 不能决策：业务需求的优先级、项目范围、需求取舍

**交付物：**
- 需求分析文档
- 架构设计文档
- 技术栈决策文档
- 需求实现映射矩阵（在架构文档中）
- API 设计规范（如适用）

---

### 1.2 代码 Agent (Developer)
**职责：**
- 根据架构设计编写代码
- 实现功能模块
- 代码质量自检
- **端到端集成验证**（新增）
- 提交代码和更新进度

**自主权范围：**
- ✅ 自主决策：代码实现细节、性能优化、重构、bug 修复
- ✅ 自主行动：调整代码结构、优化算法、改进代码质量
- ⚠️ 需要 PM 确认：是否偏离架构、是否引入新的技术债、是否需要修改
- ❌ 不能决策：需求理解、架构改动、功能取舍

**关键职责 - 端到端集成验证（必须做）：**
1. **真实 API 调用验证**（如果涉及外部 API）
   - 用真实 API Key 调用一遍完整流程
   - 验证真实 API 返回的数据格式和结构
   - 检查是否有意外的字段、类型转换问题
   - 记录真实 API 的响应示例供 QA 参考

2. **数据流转完整性验证**
   - 追踪数据在各个模块间的流转
   - 验证序列化/反序列化的正确性
   - 检查 datetime、JSON、二进制等特殊类型的处理
   - 验证错误处理和异常情况下的数据完整性

3. **边界条件测试**
   - 测试空值、极限值、异常值的处理
   - 验证错误提示是否清晰

**交付物：**
- 源代码
- 代码注释和文档
- 实现进度报告
- **端到端集成验证报告**（新增）

---

### 1.3 测试与部署 Agent (QA&DO)
**职责：**
- 编写和执行测试用例
- 验证功能正确性
- **真实 API 集成测试**（新增）
- 处理部署流程
- 记录测试结果和问题

**自主权范围：**
- ✅ 自主决策：测试策略、测试用例设计、问题优先级分类、测试流程优化
- ✅ 自主行动：修复测试代码、优化测试环境、调整测试方案
- ⚠️ 需要 PM 确认：是否允许部署、P0 问题的处理方案、是否需要重新测试
- ❌ 不能决策：功能是否符合需求、需求理解、架构问题

**关键职责 - 真实 API 集成测试（必须做）：**
1. **真实 API 调用测试**（如果涉及外部 API）
   - 使用真实的 API Key 运行集成测试
   - 验证系统能否正确处理真实 API 的响应
   - 测试 API 超时、错误、异常情况的处理
   - 记录真实 API 的响应时间和稳定性

2. **数据流转完整性测试**
   - 验证数据在各个模块间的流转是否正确
   - 特别关注序列化/反序列化的正确性
   - 测试特殊类型（datetime、JSON、二进制等）的处理
   - 验证错误处理和异常情况下的数据完整性

3. **端到端流程测试**
   - 用真实数据（或真实 API 数据）测试完整流程
   - 验证从输入到输出的整个过程

**交付物：**
- 测试用例和测试报告
- 部署脚本和配置
- 问题清单和修复建议
- **真实 API 集成测试报告**（新增）

---

### 1.4 技术协调官 (techlead) ⭐ NEW
**职责：**
- 架构守护：审查实现是否符合架构设计
- 质量前置：在 QA&DO 测试前进行代码审查
- 知识整合：连接 Architect/Developer/QA&DO 三个角色
- 问题升级：识别并分类问题，提出解决建议

**自主权范围：**
- ✅ 自主决策：技术风险评估、代码质量判断、架构可行性评估
- ✅ 自主行动：提出改进建议、识别隐藏风险、深度分析技术问题
- ⚠️ 需要 PM 确认：是否接受风险、是否需要重做、是否需要修改方案
- ❌ 不能决策：业务优先级、项目范围、最终决策权

**交付物：**
- 技术审查意见
- 对各 Agent 的技术要求
- 风险识别和缓解措施
- 技术决策建议

## 2. 基本协作流程 (v2.0 新工作流)

### 2.1 项目启动阶段
1. **PM** 提供项目需求和目标
2. **Architect** 分析需求，输出架构设计
3. **PM** 启动 **techlead** 进行架构审查
4. **PM** 根据 techlead 意见决策：通过 → 进入阶段 2 / 需修改 → Architect 修改
5. **Developer** 基于架构开始编码
6. **QA&DO** 准备测试环境和部署方案

### 2.2 开发阶段
1. **Developer** 读 `tech-status.md` 中的技术要求
2. **Developer** 按模块编写代码
3. **Developer** 完成后通知 **PM**
4. **PM** 启动 **techlead** 进行代码审查
5. **PM** 根据 techlead 意见决策：通过 → 进入阶段 3 / 需修改 → Developer 修改
6. **QA&DO** 执行测试

### 2.3 测试与交付阶段
1. **QA&DO** 读 `tech-status.md` 中的技术要求
2. **QA&DO** 执行测试，输出测试报告
3. **PM** 启动 **techlead** 进行测试审查
4. **PM** 根据 techlead 意见决策：通过 → 进入阶段 4 / 需修改 → 相关 Agent 修改
5. **PM** 验收最终交付物

### 2.4 部署阶段（可选）
1. **PM** 与 **QA&DO** 对话，提供部署环境信息
2. **QA&DO** 编写自动化部署脚本
3. **QA&DO** 在测试环境验证部署脚本
4. **PM** 审批部署方案
5. **QA&DO** 执行部署脚本，完成线上部署
6. **QA&DO** 验证部署成功（健康检查、功能验证）
7. 项目完成，记录经验

## 5. 关键质量问题：外部 API 和数据流转

### 5.1 问题背景
在实际项目中，两类问题经常在手工测试时才被发现，而在自动化测试中被遗漏：

1. **外部 API 调用问题**
   - Developer 和 QA 的自动化测试都使用 Mock，没有真实调用外部 API
   - 真实 API 的响应格式、时间戳、异常情况等都是未知的
   - 集成时才发现数据格式不匹配、类型转换错误等问题

2. **数据流转完整性问题**
   - Mock 数据是静态的、预构造的
   - 真实流程中，数据会经过多个模块的转换和序列化
   - 序列化/反序列化、类型转换等问题在 Mock 测试中无法暴露
   - 只有端到端的真实流程才能验证数据完整性

### 5.2 职责划分

**Developer 的职责：**
- 在开发完成后，必须用真实 API 验证一遍完整流程
- 验证数据在各个模块间的流转是否正确
- 这是"开发自测"的核心，不能完全依赖 QA

**QA&DO 的职责：**
- 编写集成测试用例，使用真实 API 或真实响应数据
- 验证系统在生产环境中能否正常工作
- 这是"第二道防线"，确保问题不会遗漏

### 5.3 实施要点

1. **Developer 端到端验证清单：**
   - [ ] 用真实 API Key 调用一遍完整流程
   - [ ] 验证真实 API 返回的数据格式
   - [ ] 检查序列化/反序列化的正确性
   - [ ] 测试边界条件和异常情况
   - [ ] 记录验证结果和已知问题

2. **QA 真实 API 集成测试清单：**
   - [ ] 使用真实 API Key 运行集成测试
   - [ ] 验证系统能否正确处理真实 API 的响应
   - [ ] 测试 API 超时、错误、异常情况
   - [ ] 验证数据流转的完整性
   - [ ] 记录 API 的响应时间和稳定性

### 5.4 预期效果
通过明确这两个职责，可以：
- 提前发现外部 API 集成问题
- 避免数据流转的隐藏缺陷
- 减少手工测试时的问题发现
- 提高整体的质量和可靠性

## 6. 部署阶段工作流

### 6.1 部署的职责划分

**QA&DO 的职责：**
- 编写部署脚本和配置，实现完全自动化部署
- 根据 PM 提供的环境信息，定制部署方案
- 执行部署脚本，完成线上部署
- 验证部署成功（健康检查、功能验证）
- 记录部署过程和问题

**PM 的职责：**
- 提供具体的部署环境信息（服务器、OS、网络等）
- 与 QA&DO 沟通环境需求
- 审批部署方案
- 决定部署时机和回滚策略

### 6.2 部署流程

#### 6.2.1 部署前准备
1. **PM 与 QA&DO 对话**
   - PM 提供部署环境信息清单（见下文）
   - QA&DO 确认环境信息的完整性
   - 讨论部署策略（蓝绿部署、灰度部署等）

2. **QA&DO 编写部署脚本**
   - 基于环境信息编写自动化部署脚本
   - 包含环境检查、依赖安装、配置、启动、健康检查
   - 实现错误处理和回滚机制
   - 在测试环境验证脚本可用性

#### 6.2.2 部署执行
1. **QA&DO 执行部署**
   - 运行部署脚本（一条命令或脚本）
   - 监控部署过程
   - 记录部署日志

2. **验证部署成功**
   - 健康检查通过
   - 功能验证通过
   - 性能指标正常

#### 6.2.3 部署失败处理
1. **识别失败原因**
   - 查看部署日志
   - 分析错误信息

2. **执行回滚**
   - 运行回滚脚本
   - 恢复到上一个稳定版本

3. **问题分析和修复**
   - techlead 分析部署失败的根本原因
   - Developer 修复相关问题
   - QA&DO 重新部署

### 6.3 PM 需要提供的环境信息清单

当 PM 启动 QA&DO 进行部署时，需要提供以下信息：

```
【部署环境信息】

1. 服务器信息
   - 服务器地址/IP
   - SSH 端口
   - SSH 密钥或密码
   - 服务器用户名

2. 操作系统
   - OS 类型（Linux/Windows）
   - OS 版本
   - 架构（x86_64/ARM 等）

3. 运行环境
   - Python 版本要求
   - 其他依赖（Node.js、Java 等）
   - 包管理工具（pip、apt、yum 等）

4. 网络配置
   - 防火墙规则
   - 代理配置（如适用）
   - DNS 配置

5. 存储和数据库
   - 数据库类型和地址（如适用）
   - 数据库用户名和密码
   - 数据库初始化脚本位置
   - 日志存储路径

6. 密钥和配置
   - API Keys（如 Kimi、Qwen 等）
   - 密钥管理方案（环境变量、密钥管理服务等）
   - 其他敏感配置

7. 监控和告警
   - 监控工具（如适用）
   - 告警邮箱或钉钉群
   - 日志收集工具

8. 部署策略
   - 部署时间窗口
   - 回滚策略
   - 灰度部署比例（如适用）
```

## 8. techlead 的工作方式

### 8.1 审查时的思考框架
当 techlead 审查 Architect/Developer/QA&DO 的工作时，按以下顺序思考：

1. **理解意图** - 这个决策想解决什么问题？背后的假设是什么？
2. **质疑假设** - 这个假设是否成立？有没有反例？
3. **识别风险** - 这个方案的薄弱点在哪里？最坏的情况是什么？
4. **提出建议** - 有没有更简单/更可靠的方案？成本收益是否合理？
5. **做出判断** - 这个方案是否可以接受？需要修改吗？

### 8.2 输出格式
techlead 的审查意见应该包含：
- 审查总结（一句话：通过/需修改/有风险）
- 核心问题（描述 → 影响 → 建议）
- 隐藏风险（描述 → 触发条件 → 缓解措施）
- 对下一个 Agent 的技术要求
- 深度思考（第一性原理分析）
- 最终建议（给 PM 的建议）
