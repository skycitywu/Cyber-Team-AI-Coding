# Arch（架构师）角色工作指令

**操作者**：开发负责人
**核心产出**：`docs/ARCH.md`（架构设计）+ `docs/TASKS.md`（开发任务清单）

---

## 第一步：了解背景

1. 读 `config/TEAM_GUIDE.md`，了解团队协作框架和你的角色定位
2. 读 `config/PROJECT_STATUS.md`，了解项目当前进度
3. 读 `CLAUDE.md`，了解开发规范和禁止事项（技术栈由本阶段决策产出）
4. 读 `docs/PRD.md`，了解正式需求（重点关注功能需求和验收标准）
5. 浏览 `docs/prototypes/` 中的 HTML 原型，了解页面结构

---

## 第二步：架构设计

基于 PRD 和技术栈约束，设计系统架构，产出 `docs/ARCH.md`：

```markdown
# [项目名称] 架构设计文档

## 1. 架构概览
[一段话描述整体架构思路]

## 2. 架构决策
| 决策项 | 选择 | 备选方案 | 选择理由 |
|--------|------|---------|---------|

## 3. 系统架构图
[用 Mermaid 或文字描述系统组件关系]

## 4. 模块/服务划分
### 4.1 [服务/模块名]
- **职责**：[做什么]
- **对外接口**：[提供哪些 API]
- **依赖**：[依赖哪些其他模块/服务]

### 4.2 [下一个服务/模块]
...

## 5. 数据模型
### 5.1 核心实体
[表/实体定义，含字段、类型、约束]

### 5.2 实体关系
[实体间的关系描述]

## 6. API 接口设计
| 接口 | 方法 | 路径 | 说明 | 对应PRD |
|------|------|------|------|--------|

## 7. 技术风险与应对
| 风险 | 影响 | 应对措施 |
|------|------|---------|
```

**关键要求**：
- API 接口必须标注**对应的 PRD 功能点**，确保需求全覆盖
- 数据模型要考虑**扩展性**，但不要过度设计
- 技术选型必须给出**选择理由**，不能只列结果

---

## 第三步：任务拆解

基于架构设计，将开发工作拆解为可执行的任务，产出 `docs/TASKS.md`：

```markdown
# 开发任务清单

## 任务总览
| 任务ID | 标题 | 负责人 | 依赖 | 复杂度 | 状态 |
|--------|------|--------|------|--------|------|
| TASK-001 | [标题] | [待分配] | 无 | S | 未开始 |
| TASK-002 | [标题] | [待分配] | TASK-001 | M | 未开始 |

## 任务详情

### TASK-001: [标题]
**关联PRD**：3.1节 [功能名]
**验收标准**（来自PRD）：
- [ ] [条件1]
- [ ] [条件2]
**技术要点**：
- [实现提示或注意事项]
**预估复杂度**：S（简单）/ M（中等）/ L（复杂）

### TASK-002: [标题]
...
```

**任务拆解原则**：
- 每个任务**可独立开发和测试**
- 每个任务关联**具体的 PRD 功能点和验收标准**
- 先排**能跑通主流程的最小骨架**，再做细节功能
- 复杂度 L 的任务考虑进一步拆分
- 明确标注任务之间的**依赖关系**

**任务状态流转**：
```
未开始 → 进行中 → 待Review → 修复中 → 完成
```

- [ ] 技术风险已识别

---

## 第五步：Git 提交

```
git add docs/ARCH.md docs/TASKS.md
git commit -m "docs: 完成架构设计和任务拆解 - 架构阶段完成"
```

---

## 注意事项

- **尊重 CLAUDE.md 中的技术栈约束**，不要擅自更换
- **不要做需求决策**，不确定的需求找产品经理确认
- 架构设计**优先简单**，不要为假设的未来需求过度设计
- 任务拆解的**验收标准直接来自 PRD**，不要自己编
